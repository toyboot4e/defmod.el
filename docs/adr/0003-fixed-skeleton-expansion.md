# Fixed-skeleton expansion

Every Block expands through one hardcoded template: a strict single-pass parser fills named Slots (ensure, init, load form, config), and a visible backquote/pcase assembles them. We rejected the leaf/use-package engine — recursive keyword handlers that each receive the expansion of the remaining keywords and decide to wrap or append, requiring a canonical keyword-sort pass — because that engine is exactly the part of leaf we found inscrutable: handlers are eval'd s-expressions over dynamic variables, output order is reverse execution order, and the eager/deferred split is encoded in table position. The cost we accepted: a new keyword with novel wrapping semantics requires editing the template itself; the DSL is not open for third-party keyword extension. For a personal DSL whose expansion runs uncompiled at every startup, a closed, legible, single-pass engine is the point.
